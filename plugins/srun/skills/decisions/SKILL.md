動手前的決策收斂閘門。explore 把方案聊開（發散），這個 skill 反過來把使用者收斂到明確（收斂）——沿著設計的決策樹走，只挑出「未定且必須定」的分支問使用者，避免模糊在 propose 被包裝成看似完整的 spec、最後在 Coder / Reviewer / retry 階段才爆出來。

定位：`opsx:explore` 之後、`opsx:propose` 之前。**只收斂決策、不產 spec、不寫 code。**

**Input**：對話中已有 explore 結論或一段需求描述。

---

## 適用判斷

先確認這個任務值得跑。判斷軸不是「前端 vs 後端」，是**有沒有未定的決策分支**：

| 情境 | 用 | 跳過 |
|------|----|------|
| 完整新功能（牽涉資料 / 邏輯 / 多模組） | ✓ | |
| 全新 UI 流程設計（空狀態 / 載入 / 錯誤 / 邊界互動待定） | ✓ | |
| 既有功能的調整、需求已明確 | | ✓ |
| 純樣式微調、文案、單行修正（Tier 1） | | ✓ |

一句話自問：**「這任務有沒有『使用者還沒定、但 Coder 動手時必須有答案』的分支？」** 有就跑，沒有就直接 `opsx:propose`。

---

## 核心判準（這是規範，不是清單）

把設計當成一棵**決策樹**：沿著分支走，逐一解決分支間的依賴。一個「待決策分支」需同時成立：

1. **存在多種合理實作**：同一個需求底下有不只一種做法，且選哪種會改變使用者看到的行為或資料結構。
2. **目前沒有明確答案**：explore 結論、CLAUDE.md、既有慣例、現有程式碼都推不出來。

任一不成立就**不是**待決策分支，跳過：
- 只有一種合理做法 → 不算。
- 答案能從 CLAUDE.md / 既有慣例 / 程式碼讀出來 → **自己讀，不要問使用者**。
- 純風格、不影響行為 → 不算。

決策樹是你自己長出來的：依**本任務的具體性質**推導分支，不要被下方引子侷限。

---

## 起頭引子（常見分支類別，舉例，非窮舉）

只是幫你起頭的種子，不是要逐項填答的表單：

- 邊界值 / 極端輸入
- 空狀態 / 無資料
- 錯誤 / 失敗情境的行為
- 載入中 / 非同步的中間狀態
- 需求之間衝突時以哪個為準
- 權限 / 身分差異

> 用這幾類起頭，再依任務性質主動延伸。**清單外的分支同樣要找；清單內不相關的直接略過。**

## 流程

### Step 1：建決策樹

讀 explore 結論（或需求描述）+ 專案 CLAUDE.md + 相關既有程式碼，依「核心判準」列出所有待決策分支。

- **能查 codebase / CLAUDE.md / 既有慣例就先查**，能推出答案的分支直接標記為「已定（依據：…）」，不拿來問人。
- 宣告：「srun:decisions：{任務摘要}，找到 N 個待決策分支」。
- **找不到任何待決策分支時，直說「無未定決策，可直接 propose」，不要硬湊問題。**

### Step 2：逐一收斂（一次只問一個）

**一次只問一個問題，等使用者回答後再問下一個。一次丟一堆問題會讓人當機。**

每個問題：
1. 點出這是哪個分支、為什麼需要現在定（影響什麼行為 / 資料）。
2. **附上你的建議答案與理由**——讓使用者多數時候只要點頭。
3. 若這個答案會牽動其他分支（依賴），在使用者回答後據此調整後續問題，不要問已被前一個決策蓋掉的問題。

使用者可能回答、可能改你的建議、也可能說「這個你決定」——尊重使用者的選擇，記下結論。

### Step 3：輸出決策清單

所有待決策分支都有答案後，輸出一份結構化決策清單（格式見下），供使用者接著跑 `opsx:propose`。

**不執行 propose、不產 spec、不寫 code。** 收斂到此為止。

---

## 輸出格式

```
## 決策清單：{任務摘要}

### 已收斂決策
| # | 決策點 | 結論 | 依據 / 理由 |
|---|--------|------|------------|
| 1 | {分支} | {選定做法} | {使用者的回答 / 自查依據} |
| 2 | ... | ... | ... |

### 自查已定（未打擾使用者）
- {分支} → {結論}（依據：CLAUDE.md / 既有慣例 / 程式碼）

### 仍開放（使用者選擇暫不定 / 留給實作）
- {分支}（原因：{使用者說由 Coder 決定 / 待 propose 細化}）

### 下一步
決策已收斂，可執行 `opsx:propose`。執行 propose 時，將本清單「依據 / 理由」欄寫入 design.md 的決策記錄；「仍開放」項目在 design.md 標示為**刻意留給 Coder 的留白**（非遺漏），避免下游誤補或誤刪。
```

無待決策分支時，只輸出：

```
## srun:decisions：{任務摘要}
無未定決策（依據：explore 結論已明確 / 可由既有慣例推得）。可直接執行 `opsx:propose`。
```

## Guardrails

- 只收斂決策，**不產 spec、不寫 code、不跑 propose**——職責單一。
- 一次只問一個問題，附建議答案；不要一次丟整批問題。
- 能從 CLAUDE.md / 既有慣例 / 程式碼推出的，自己查，不打擾人。
- 引子清單是種子不是規格：依任務延伸分支，不被清單侷限，也不為填滿清單硬湊問題。
- 找不到待決策分支就如實說「無未定決策」，不製造假問題。
- 大方向通常沒問題，目標是攔住會洩漏的小細節——保持輕量，掃完未定分支即止，不做完整設計逼問。
